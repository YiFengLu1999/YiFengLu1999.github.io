# Why Machines Learn 博文 Markdown 撰写规范

这份规范适用于 `Why Machines Learn` 系列读书笔记，目标是让每篇文章在 Jekyll 博客中保持统一的结构、排版风格和数学公式显示效果。

## 1. 文件命名规范

- 文件放在 `_posts/` 目录下。
- 文件名使用日期加标题的形式：

```text
YYYY-MM-DD-Why-Machines-Learn-第X章读书笔记.md
```

- 示例：

```text
2026-03-15-Why-Machines-Learn-第二章读书笔记.md
```

## 2. Front Matter 规范

每篇文章开头必须使用 YAML front matter，字段顺序尽量保持一致。

```yaml
---
layout:     post
title:      Why Machines Learn 第X章读书笔记：一句明确的主标题
subtitle:   用一句话概括本章的核心问题、直觉或冲突
date:       YYYY-MM-DD
author:     YifengLu
header-img: img_Blog/xxx.jpg
catalog:    true
tags:
    - 读书笔记
    - 机器学习
    - 主题词1
    - 主题词2
    - 主题词3
---
```

### 字段说明

- `layout`: 固定为 `post`
- `title`: 采用“书名 + 章节 + 文章主标题”的格式
- `subtitle`: 用一句话概括文章核心内容，偏解释性，不重复 `title`
- `date`: 与文件名前缀一致
- `author`: 作者名，单人或多人均可，但建议格式统一
- `header-img`: 使用 `img_Blog/` 下的封面图
- `catalog`: 固定为 `true`
- `tags`: 建议 4 到 6 个，兼顾主题和方法

## 3. 正文结构规范

正文从 front matter 之后直接开始，不要把整篇文章包在代码块里。

### 3.1 标题层级

- 文章正文的主章节使用一级标题 `#`
- 主章节内部的小节使用二级标题 `##`
- 一般不建议无必要地继续下钻到三级标题 `###`

推荐结构：

```markdown
# 开篇问题或导引

# 第一部分主线

## 其中一个子问题

# 第二部分主线
```

### 3.2 段落与分隔

- 章节之间使用 `---` 做分隔
- 段落之间空一行
- 只有在确实需要强制换行时，才在行尾保留两个空格

### 3.3 风格建议

- 开头先抛出问题或核心判断
- 中间按“概念 -> 直觉 -> 例子 -> 推论”展开
- 结尾用回顾、判断或尾声收束
- 可以保留有个人风格的表达，但整体结构要清楚

## 4. 常用 Markdown 写法

### 4.1 强调

- 重点判断用加粗：`**重点内容**`
- 书名可以写成：`《**Why Machines Learn**》`

### 4.2 引用

重要观点、结论、定义，优先使用引用块：

```markdown
> **这是文章里最关键的一句话。**
```

### 4.3 列表

- 并列信息用无序列表
- 过程、步骤、推演用有序列表

示例：

```markdown
- 第一层意思
- 第二层意思

1. 第一步
2. 第二步
3. 第三步
```

## 5. 数学公式规范

这是本系列最重要的排版约束之一。

### 5.1 行内公式

行内数学统一使用单个美元符号：

```markdown
$w$, $x$, $\theta$, $w^T x$, $y \in \{+1,-1\}$
```

示例效果：

- $w$
- $x$
- $\theta$
- $w^T x$
- $y \in \{+1,-1\}$

### 5.2 块级公式

单独成行的公式统一使用：

```markdown
$$
w \leftarrow w + yx,\quad b \leftarrow b + y
$$
```

不要使用以下写法：

- `\[` `\]`
- 整篇文章外层套代码块
- 用反引号包裹公式

### 5.3 数学与代码的边界

- 数学符号用 `$...$` 或 `$$...$$`
- 普通术语、变量名示例、英文关键词可用反引号，如 `token`、`query`、`key`
- 不要把本该是数学公式的内容写成反引号代码

正确示例：

```markdown
给你一个数据点 $x$，再给你一个权重向量 $w$，算一下 $w^T x$。
```

不推荐示例：

```markdown
给你一个数据点 `x`，再给你一个权重向量 `w`，算一下 `w^T x`。
```

## 6. 内容组织建议

建议每篇文章尽量包含以下几类内容中的大部分：

- 这一章在讲什么
- 核心主线或核心矛盾是什么
- 一个或多个关键概念的解释
- 数学公式背后的几何直觉
- 模型为什么成立，或为什么会失败
- 最后的总结、判断或前瞻

适合本系列的表达方式：

- 少堆砌公式，多解释公式“在空间里做了什么”
- 少纯定义，多用类比、直觉、图景化语言
- 强调“模型行为 = 几何动作 / 优化过程 / 结构限制”

## 7. 不要出现的问题

- 不要把整篇正文包进代码块
- 不要残留 `contentReference`、`oaicite` 之类的引用污染
- 不要混用 `$...$` 和 `\[` `\]` 作为显示公式风格
- 不要缺少 front matter
- 不要让文件名日期和 front matter 的 `date` 不一致
- 不要使用和正文无关的封面图路径

## 9. 发文前自检清单

发布前建议逐项检查：

- front matter 是否完整
- `title`、`subtitle`、`date`、`header-img` 是否已填写
- 文件名日期是否和 `date` 一致
- 一级标题和二级标题层级是否清楚
- 章节之间是否使用了 `---`
- 数学是否统一使用 `$...$` 和 `$$...$$`
- 是否还有 `contentReference` 或其他脏标记
- 是否误把整篇文章放进代码块
- 封面图路径是否可用
- tags 是否和本章主题匹配

## 10. 一句话原则

这类博文不是在堆知识点，而是在把“数学对象如何运动、如何分界、如何失败”写成一篇读起来顺手的解释性文章。
